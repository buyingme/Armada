# Fleet Builder Requirements

Status: FB0 baseline
Last updated: 2026-05-26

## 1. Purpose

The fleet builder is an integrated local-first tool for creating, validating,
saving, loading, importing, exporting, and preparing Star Wars: Armada fleets for
play inside this Godot project.

The builder must not be an isolated list editor. Its roster and setup contracts
must support hot-seat setup, network setup, save/load, replay, and later backend
sync without introducing a second fleet format.

## 2. Scope Decisions

| Area | Requirement |
|---|---|
| MVP storage | Fleets are stored locally first through versioned JSON. |
| MVP exchange | Native JSON import/export is required. PDF, CSV, and external-builder imports are deferred. |
| MVP formats | Standard 400, Core Set 180, and custom agreed point totals are supported. |
| MVP catalog | Core Set ships, squadrons, upgrades, objectives, and obstacles are the first completeness target. Existing Wave 1 generic Imperial squadrons may stay in the catalog, but they are not the Core Set gate. |
| MVP setup | Two valid fleets can be consumed by setup, objective choice, obstacle placement, deployment, and match start. |
| Network setup | Network games must consume the same setup package contract as local setup. The host validates the expanded package, and both peers agree on the same package hash before match start. |
| Rules reference | Generic rules and component-specific rules are browsable through a rules-reference catalog. Generic rules have one source of truth for display text and implementation tracking. |
| Backend | Accounts, cloud sync, remote fleet storage, APIs, and share links are deferred infrastructure. Deferring backend work must not defer network game setup. |

## 3. MVP Functional Requirements

| ID | Requirement |
|---|---|
| FB-REQ-001 | Users can create, rename, duplicate, delete, save, and load local fleets. |
| FB-REQ-002 | A fleet has a name, description, faction, point format, point limit, ships, squadrons, upgrades, and three objectives. |
| FB-REQ-003 | Users can add, edit, and remove ships, squadrons, ship upgrade assignments, and objective selections. |
| FB-REQ-004 | The builder continuously calculates total fleet points, ship points, squadron points, and upgrade points. |
| FB-REQ-005 | The catalog supports search and filtering by faction, component type, point cost, upgrade slot or type, wave, expansion, keyword, rules-reference category, implementation status, and search tags. |
| FB-REQ-006 | The builder validates point total, faction alignment, commander/flagship rules, squadron point cap, unique-name limits, unique squadron limits, objective category selection, and upgrade assignment legality. |
| FB-REQ-007 | Validation results are structured and deterministic, including severity, rule id, source reference, affected roster entry ids, and readable user-facing text. |
| FB-REQ-008 | Users can import and export the native versioned fleet roster JSON. |
| FB-REQ-009 | The local library stores multiple versions or snapshots per fleet and can restore an older version. |
| FB-REQ-010 | Two valid rosters can be expanded into a match-ready setup package that embeds both rosters and has no dependency on local fleet-library files. |
| FB-REQ-011 | Setup records first player, selected objective, objective owner, objective chooser, obstacles, deployments, and any objective-derived setup state in JSON-safe fields. |
| FB-REQ-012 | Setup and deployment positions serialize as normalized `pos_x`, `pos_y`, and `rotation_deg`, never pixels or Godot value types. |
| FB-REQ-013 | Save/load after setup preserves selected objective, obstacle placements/state, deployments, fleet points, ships, squadrons, upgrade assignments, and objective-derived runtime state. |
| FB-REQ-014 | The fleet-builder UI is integrated in the Godot project and follows the existing game styling. UI renders core catalog, roster, validation, and library results; it does not own rules. |
| FB-REQ-015 | The UI provides a Rules Reference section for generic rules and component-specific rules, including visible implementation status for pending, partial, and integrated gameplay rules. |
| FB-REQ-016 | Component entries can open linked rules-reference records from catalog search results or roster selections. |
| FB-REQ-017 | Network lobbies consume one expanded roster payload from the host and one from the client through the same setup package contract used by local setup. |
| FB-REQ-018 | Before a network match starts, the host validates the setup package and both peers confirm the same canonical package hash. |
| FB-REQ-019 | Any objective, upgrade, obstacle, squadron ability, or setup effect that becomes live is implemented through the command and `RuleRegistry` architecture so hot-seat, save/load, replay, and network games derive the same legality, UI eligibility, and state changes. |
| FB-REQ-020 | The builder remains responsive for large future fleets by keeping catalog queries deterministic, cached where useful, and independent from scene-tree work. |

## 4. Deferred Requirements

| ID | Requirement | Deferred until |
|---|---|---|
| FB-DEFER-001 | User accounts and authentication. | Native roster JSON and local setup packages are stable. |
| FB-DEFER-002 | Remote fleet library, cloud storage, and sync conflict handling. | Local library, import/export, and setup contracts are stable. |
| FB-DEFER-003 | Public or private share links/codes. | Backend direction is selected. |
| FB-DEFER-004 | PDF and CSV export. | Native JSON export is proven. |
| FB-DEFER-005 | External fleet-builder import adapters. | Native schema has stopped changing. |
| FB-DEFER-006 | Full non-Core card catalog. | Core Set validation and setup flow are proven. |
| FB-DEFER-007 | Strict tournament automation and full competitive policy packs. | Core rules and custom agreed totals are stable. |

Deferred backend and sharing work must use the same versioned roster contract as
the local builder. It must not introduce a second source of truth.

## 5. Non-Goals For MVP

- No backend dependency for creating, editing, saving, importing, exporting, or
  using fleets in local or network setup.
- No live gameplay implementation for objectives, upgrades, obstacles, or named
  ace abilities until their `RuleRegistry` slices are explicitly scheduled.
- No UI-specific enforcement of fleet or setup legality. All legality decisions
  come from core validators and static catalog data.
- No storage of pixels, `Vector2`, `Color`, resources, scene nodes, peer ids, or
  local file paths in roster or setup package JSON.
- No PlayMode-specific fleet-builder or setup logic under `src/scenes/` or
  `src/ui/`.

## 6. Data Source Policy

| Source | Role |
|---|---|
| `Resources/SWM-RULES-REFERENCE-GUIDE-150/` | Primary local rules reference for construction, setup, objective, obstacle, command, attack, defense token, scoring, and keyword rules. |
| `Resources/SWM01-ARMADA-LEARN-TO-PLAY/` | Local learning and setup reference, especially for Core Set assumptions. |
| `Resources/Game_Components/*/*_rules.txt` and `obstacles_specs.txt` | Component-level source notes used to create structured catalog JSON and rules-reference records. |
| `Resources/Game_Components/wave_expansion_set_content_information.txt` | Local wave, expansion, and Core Set inventory reference. |
| `Resources/Game_Components/*.json` | Static catalog source once normalized and validated. Runtime code reads card facts from these files, not hardcoded GDScript values. |
| `Resources/Game_Components/rules/` | Planned static rules-reference catalog for display text, source refs, tags, and implementation status. |
| `RuleRegistry` rule files | Gameplay implementation authority for live effects. Rules-reference JSON is not executable code. |
| External wiki pages | Manual verification only. External wiki data must be copied into local source notes or structured JSON before implementation depends on it. |

## 8. Construction And Validation Rules For MVP

Validators must cite the source rule in public API doc comments when they are
implemented. The MVP validation surfaces are:

| Rule area | Requirement |
|---|---|
| Point total | A fleet cannot exceed the agreed point total. Standard is 400; Core Set mode is 180; custom agreed totals are allowed. |
| Faction alignment | A fleet has one faction, and cards aligned with another faction are illegal unless a future card explicitly permits it. |
| Commander and flagship | A fleet has exactly one commander and one flagship when commanders are included. |
| Squadron cap | Squadron points cannot exceed one third of the agreed fleet total, rounded up. |
| Unique squadrons | A fleet can include one unique squadron with defense tokens per 100 points of agreed fleet total. |
| Unique names | A fleet cannot contain more than one copy of the same unique name. |
| Objectives | A fleet selects exactly one Assault, one Defense, and one Navigation objective. |
| Upgrade slots | A ship can equip only upgrades allowed by its upgrade bar and slot groups. |
| Duplicate ship upgrades | A ship cannot equip more than one copy of the same upgrade card. |
| Upgrade restrictions | Faction, size, ship trait/name, title, commander, modification, flagship, and other printed restrictions are enforced from catalog data. |

## 9. Setup And Network Contract Requirements

- The editable roster model is not runtime `PlayerState`.
- The match-ready setup package embeds two full roster payloads and does not
  depend on local fleet ids or local library files.
- Core setup JSON stores `player_index` and owner fields. Transport peer ids are
  mapped outside core setup data.
- The canonical package hash excludes local-only timestamps, local file paths,
  UI state, and other non-gameplay metadata.
- The same setup package is the handoff contract for hot-seat, network, replay,
  save/load, and bootstrap.
- If placement or objective setup happens before bootstrap, it mutates the
  setup package. If it happens after `GameState` exists, it uses commands.
- Runtime state introduced by setup must serialize and deserialize in the same
  implementation slice that introduces it.

## 10. MVP Assumptions

| Topic | MVP assumption | Later option |
|---|---|---|
| First player | First player is selected manually during setup and recorded in the setup package. | Bid/initiative automation can be added after validation and setup packages are stable. |
| Objective choice | The selected objective belongs to the second player's roster and records its owner and the choosing player in setup JSON. | Richer objective-choice UI can be added after `ObjectiveData` and setup state exist. |
| Deployment | Deployment begins as warning-guided manual placement with normalized positions. | Strict deployment-zone and overlap enforcement can harden in later setup validators. |
| Obstacles | Core Set obstacle tokens are the first setup target. | Expansion/objective-specific obstacle additions follow after obstacle JSON exists. |
| Local fleet files | The exact storage folder is a later local-library decision. | It may be project/user data or a dedicated `fleets/` folder, but setup packages cannot depend on it. |

## 11. UI Requirements

- The first screen of the fleet builder is the usable builder experience, not a
  marketing or explanatory page.
- The interface provides fleet header/edit controls, point totals, validation
  status, catalog search/filter, selected ships and upgrade slots, squadron list,
  objective selectors, local library actions, import/export, and rules-reference
  browsing.
- The UI must be efficient for repeated editing: filters are quick to adjust,
  validation messages are tied to affected roster entries, and saving/exporting
  is explicit.
- UI controls call core catalog, roster, validator, setup, and library services.
  They do not duplicate construction rules or gameplay rule logic.

## 12. FB0 Exit Criteria

- Requirements are split into MVP, deferred, and non-goal sections.
- Data-source policy identifies local authoritative sources and the limited role
  of external wiki data.
- Source inventory records the current Core Set catalog state and gaps.
- Network setup is a first-class requirement, separate from deferred backend
  infrastructure.
- First-player and deployment assumptions are explicit.
- The next implementation step is FB1: component folder and schema contract.
